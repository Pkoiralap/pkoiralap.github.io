Portfolio Architecture & Engineering Document

Sep 2023

I engineered this platform as a zero-dependency, vanilla Single Page Application (SPA). My goal was to create a "living" portfolio where each project isn't just a static image, but an isolated, interactive micro-site that exists seamlessly within the larger shell.

1. Architectural Philosophy & Strategy

When I started building this, I wanted to balance simplicity with raw technical power. This led me to three distinct content strategies:

1.1 Blogs: Keeping it Simple

I thought blogs are just blogs. They are meant for reading (if anyone decides to), so I kept them simple and used markdown (MD) files for blogs. They are easy to write, easy to parse, that's it.

1.2 Events: "Website inside of a Website" Experiment

For the timeline events, I wanted to push the boundaries. I asked myself: Can I render an entire website inside another website? and thus, I started experimenting with a mounting system that could fetch isolated HTML, CSS, and JS components and inject them into a sandboxed div. I used the Events section as my testing ground to see if I could handle dynamic style scoping and script execution without breaking the global state.

1.3 Projects: A Successful Evolution

The experiment worked! Once I saw it working, I brought that same technologia (I watch too many memes) to the Projects section. This is why projects are fully isolated HTML, CSS, and JS packages. It allows me to add cool Javascripts and custom CSS to better showcase the unique engineering of each project. For example a D3.js architecture diagrams or interactive simulations. How cool is that? If I ever create a game, you’ll be able to play it right here on my portfolio site, fully sandboxed.

Although, since the experiment worked, I might also move blogs to this format too, just to add more coolness factor. Eventually...

2. Core Architectural Components

2.1 The Mounting Layer (index.html)

The entrypoint is a lightweight HTML5 shell. It avoids build steps by resolving dependencies via native ES Modules. It loads the vintage typography stylesheet, imports Tailwind-free structural styling, and defines the target mounting div:

<div id="app"></div>

2.2 The Custom Engine (js/utils.js)

This module acts as the core controller. When a dynamic component requests resources, `utils.js` triggers a three-part fetching pipeline:

  • HTML Aggregator: Fetches and parses `main.html` into a document fragment, strips scripts/stylesheets to protect against runtime double-evaluation, and returns clean HTML content.
  • CSS Sandbox Scoper: Scopes subfolder stylesheets by parsing all rules and prefixing them with a unique container selector matching the parent ID (e.g., `#project-portfolio-site`). This blocks sub-component styles from polluting the global workspace scope.
  • JS Executor: Safely runs isolated script modules (`main.js`) in the context of the mounted component using safe evaluation sandboxes.

2.3 Scroll-Spy & Timeline (js/components/Timeline.js)

The timeline implements an active vertical tracking system. The left column contains a timeline line with node indicators. The right column renders the full event details loaded from event directories. The scrolling state is synchronized via two mechanisms:

  • Passive Scroll Spy: A scroll event listener tracks the bounding client rect of all detail cards, measuring distance relative to 30% of the viewport height. The closest card is instantly marked active.
  • Bottom-of-Page Protection: A threshold trigger detects when the user scrolls to the absolute bottom of the document and locks the active state to the oldest milestone, resolving native layout limits.

2.4 Projects Loader (js/components/Projects.js)

Manages listing and dynamic view transition for projects. Utilizing hash-routing parameters (e.g., `#/projects/portfolio-site`), it triggers the asset collection routine from the respective project subdirectory and scopes styles using modern CSS `::marker` lists for clean typography alignment.

3. Design System & Aesthetics

The website is styled using a "vintage print" design system, mimicking mid-century publication layouts with ink-on-paper textures and flat layouts.

Token Name Value / Rule Description
Background #fdfdfc (Parchment) Soft, low-contrast warm white to simulate vintage paper.
Primary Ink #8c4327 (Terracotta) Warm earth-red accent ink, used for headings, dates, and borders.
Secondary Ink #5c554c (Charcoal) Dark gray ink for meta text, labels, and secondary tags.
Body Copy #3c3530 (Carbon) Dark, high-readability text color for core body reading text.
Serif Font 'Fraunces', Georgia, serif Used for primary page titles, card headers, and project features.
Mono Font 'IBM Plex Mono', monospace Used for badge metadata, timestamps, code snippets, and stamps.
Borders 1px solid #d8d3c5 Thin, warm-gray print borders bounding cards and buttons.
