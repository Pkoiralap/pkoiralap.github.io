Fire Simulator: Particle Dynamics & 3D Engine

Jun 2026

This project is a high-performance 3D fire simulation engine built from scratch using C++ and SDL. It implements a custom particle system with physical properties like velocity, lifetime, and intensity, rendered through a home-grown 3D rasterization pipeline featuring Z-buffering and geometric transformations.

1. System Architecture

The engine is architected as a modular C++ system where core mathematical utilities support a specialized particle engine and a custom 3D renderer. Below is an interactive flow diagram rendering the architectural dependencies and data flows of the simulator.

2. 3D to 2D Projection Mathematics

The engine converts 3D world coordinates to 2D screen pixels using a perspective projection pipeline. This process involves shifting the world coordinate system to the camera's perspective (View Space) followed by a non-linear projection.

2.1 View Transformation

The engine constructs a World-to-Camera (WtoC) matrix using the N, U, V basis vectors:

  • N (Forward): Normalized vector from LookAt to Camera.
  • U (Right): Cross product of Up and N.
  • V (Up): Cross product of N and U.

The final transformation is the product of a Rotation matrix R and a Translation matrix T: WtoC = R * T.

2.2 Perspective Projection

Once in Camera Space, the 3D point $(x, y, z)$ is projected using the $z$-coordinate to simulate depth. The logic in Transformation.cpp follows:

screen_x = (view_x / -view_z + planeWidth / 2) / planeWidth * winWidth
screen_y = (1 - (view_y / -view_z + planeHeight / 2) / planeHeight) * winHeight

Dividing by $-z$ ensures that objects further away (larger negative $z$) appear smaller and closer to the screen center.

3. The Particle Engine & Dynamics

The core simulation engine (Fire.cpp) manages thousands of individual particles to create the rising fire effect. The motion is governed by a combination of thermal buoyancy and stochastic variance.

3.1 Rising Motion (Buoyancy)

Particles simulate the upward draft of heat. Each frame, a particle's vertical position is updated based on its intensity i. Brighter, "hotter" particles rise faster, creating a tapered flame shape:

vel.y = 20 * particle.intensity; pos = pos + vel;

3.2 Sinusoidal Flicker

To mimic the natural "lick" and sway of a flame, the engine applies a horizontal sinusoidal displacement based on the particle's index i. This ensures that the fire doesn't just rise in a straight line but moves with organic turbulence:

vel.x = sin(index) * 10 * random_variance;

3.3 Intensity Decay & Lifetime

As particles rise, they lose energy. The intensity i is decremented every frame. When i falls below a threshold or the lifetime expires, the particle is recycled. This manages the flame's height and ensures a smooth "fading out" effect at the tip of the fire.

4. Lighting & Gouraud Shading Implementation

To achieve smooth lighting across 3D surfaces like the teapot, the engine implements Gouraud Shading. This method calculates light intensity at vertices and interpolates them across the triangle face.

4.1 Normal Averaging

For smooth surfaces, the engine first calculates the face normal for every triangle and then averages them at each vertex to get the Vertex Normal. This prevents "faceted" looks where each triangle is a solid color.

4.2 Vertex Intensity Calculation

Intensity is calculated using the Lambertian Reflectance model. For each vertex, the engine computes the dot product between the vertex normal and the light vector (provided by the fire particles):

Intensity = Normal • (Light_Pos - Vertex_Pos).normalized()

This determines how "bright" each vertex is based on its orientation relative to the fire.

4.3 Linear Interpolation (The Scanline)

During rasterization (drawSpan), the engine linearly interpolates the intensity values from the three vertices across the entire face, resulting in the smooth color transitions seen in the demo.

5. Custom 3D Object Loader (.obj)

The engine features a hand-written wavefront .obj parser in Object.cpp. This loader is responsible for translating raw geometric data into the engine's internal Object3d structures.

5.1 Mesh Parsing & Geometry

The loader parses v (vertices), vn (normals), and f (faces) tokens. It maintains a Vertex Buffer and a Surface Buffer. Because raw .obj files often lack smooth normal data, the engine implements a post-loading Normal Averaging algorithm:

Normal_Vertex = Σ(Face_Normals_Connected_to_Vertex) / Count

This allows the Object3d class to provide the smooth vectors required for Gouraud shading.

5.2 Relationship to Rendering

The loader acts as the bridge between disk assets and the Transformation module. Once loaded, every vertex in the buffer is transformed by the camera matrix and projected to 2D screen space using the Z-buffer logic described previously.

6. Fire Particles as Dynamic Light Sources

One of the most advanced features of the engine is that the fire is not just a visual effect; it acts as a dynamic light source for other 3D objects in the scene.

6.1 Particle Sampling

During the render() call of a 3D object, the engine samples the Fire system. Instead of using a single global light, it iterates through a subset of active fire particles to calculate the illumination at each vertex:

for(fireParticle p : FireSystem) {
    Vec3 LightDir = (p.pos - vertex.pos).normalized();
    intensity += vertex.normal • LightDir;
}

6.2 Dynamic Intensity & Flicker

Because fire particles are constantly moving and dying, the light vectors change every frame. This creates a natural flicker effect on the surfaces of the teapot and torch without any hard-coded lighting animations. The intensity is further scaled by the particle's age, ensuring that brighter, "younger" particles contribute more to the scene's illumination.

7. Visibility & Z-Buffer Depth Sorting

To correctly handle occlusion (showing which object is in front of another), the engine uses a Z-Buffer (Depth Buffer).

7.1 The Depth Test

A 2D array Zbuffer[width * height] stores the $z$-value of the closest pixel rendered at every screen coordinate. During the setpixel call, the engine performs a depth test:

if (P.z > Zbuffer[x][y]) return; // Current pixel is behind what's already drawn
else Zbuffer[x][y] = P.z;       // Current pixel is closer, update buffer and draw

This ensures that the teapot doesn't "peek through" the torch if it's actually behind it.
